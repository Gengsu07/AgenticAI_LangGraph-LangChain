# 8. How LangChain Agents Understand & Call Tools Automatically

## 🎯 Core Question
> "How can an agent understand the city asked by the user, then automatically use it as an argument to `get_weather()`?"

---

## 📌 Short Answer

The LLM **does NOT literally parse** the sentence with regex or string matching.  
Instead, the LLM uses **Tool Calling** — a built-in capability where the model:
1. Reads the **tool schema** (name, description, parameter types)
2. Understands the **user's intent** from natural language
3. Generates a **structured JSON** with the function name + extracted arguments
4. The framework then **executes** the function with those arguments

---

## 🧠 The Complete Flow (Step by Step)

### Step 1: Define the Tool with Schema

```python
from langchain.tools import tool

@tool
def get_weather(location: str) -> str:
    """Get the weather at a location."""
    return f"It's sunny in {location}"
```

**What happens behind the scenes:**
The `@tool` decorator converts your function into a **Tool object** with:
- **Name**: `"get_weather"` (from function name)
- **Description**: `"Get the weather at a location."` (from docstring)
- **Schema**: `{"location": {"type": "string"}}` (from type hints)

This schema is sent to the LLM as part of the prompt, so the LLM **knows** what tools are available and what arguments they expect.

### Step 2: Bind Tools to the Model

```python
from langchain.chat_models import init_chat_model

model = init_chat_model("groq:qwen/qwen3-32b")
model_with_tools = model.bind_tools([get_weather])
```

`bind_tools()` tells the model: *"Hey, you have these tools available. If the user asks something that needs a tool, generate a tool call instead of a text response."*

### Step 3: User Sends a Message

```python
response = model_with_tools.invoke("What's the weather like in New York?")
```

### Step 4: LLM Analyzes and Generates Tool Call

The LLM receives a prompt like this internally:

```
System: You have access to the following tools:
  - get_weather(location: str): Get the weather at a location.

User: What's the weather like in New York?
```

The LLM's reasoning process:
1. ✅ The user is asking about weather → I have a `get_weather` tool!
2. ✅ The tool needs a `location` parameter (type: string)
3. ✅ The user mentioned "New York" → that's the location
4. → Generate a **tool call**: `get_weather(location="New York")`

**The LLM outputs structured JSON** (not a text response):

```json
{
  "name": "get_weather",
  "args": {"location": "New York"},
  "id": "call_abc123"
}
```

### Step 5: Framework Executes the Tool

```python
# The response contains tool_calls
print(response.tool_calls)
# [{'name': 'get_weather', 'args': {'location': 'New York'}, 'id': 'call_abc123'}]

# Execute the tool
tool_result = get_weather.invoke(response.tool_calls[0])
# ToolMessage(content="It's sunny in New York")
```

### Step 6: Pass Result Back to LLM for Final Answer

```python
messages = [
    {"role": "user", "content": "What's the weather in New York?"},
    ai_msg,          # Contains the tool call
    tool_result,     # Contains the tool execution result
]
final_response = model_with_tools.invoke(messages)
print(final_response.text)
# "The weather in New York is sunny."
```

## ⚠️ Fixing Your Original Code

Your original code has some issues:

```python
# ❌ WRONG - create_agent doesn't exist in LangChain
from langchain.agents import create_agent

# ❌ WRONG - plain function, not a LangChain tool
def get_weather(city: str) -> str:
    """Get the weather for a city."""
    return f"The weather in {city} is sunny."
```

### ✅ Correct Approaches

#### Approach 1: Manual Tool Execution Loop (Recommended for Learning)

```python
from langchain.chat_models import init_chat_model
from langchain.tools import tool

# Step 1: Define tool with @tool decorator
@tool
def get_weather(city: str) -> str:
    """Get the weather for a city."""
    return f"The weather in {city} is sunny."

# Step 2: Initialize model and bind tools
model = init_chat_model("groq:qwen/qwen3-32b")
model_with_tools = model.bind_tools([get_weather])

# Step 3: Send user message
messages = [{"role": "user", "content": "What is the weather like in New York?"}]
ai_msg = model_with_tools.invoke(messages)
messages.append(ai_msg)

# Step 4: Execute tool calls
for tool_call in ai_msg.tool_calls:
    tool_result = get_weather.invoke(tool_call)
    messages.append(tool_result)

# Step 5: Get final response
final_response = model_with_tools.invoke(messages)
print(final_response.text)
# Output: "The weather in New York is sunny."
```

#### Approach 2: Using `create_react_agent` (Automatic Loop)

```python
from langchain.chat_models import init_chat_model
from langchain.tools import tool
from langgraph.prebuilt import create_react_agent

@tool
def get_weather(city: str) -> str:
    """Get the weather for a city."""
    return f"The weather in {city} is sunny."

# create_react_agent handles the entire tool execution loop automatically
model = init_chat_model("groq:qwen/qwen3-32b")
agent = create_react_agent(
    model=model,
    tools=[get_weather],
    prompt="You are a helpful assistant."
)

response = agent.invoke({
    "messages": [{"role": "user", "content": "What is the weather like in New York?"}]
})

# Print the final message
print(response["messages"][-1].content)
```

## 🔑 Key Concepts

### Why `@tool` Decorator is Required

| Without `@tool` | With `@tool` |
|---|---|
| Plain Python function | LangChain Tool object |
| No schema generated | JSON schema auto-generated |
| LLM doesn't know about it | LLM can see name, description, args |
| Can't be bound to model | Can be bound with `bind_tools()` |

### Why Docstrings Matter

```python
@tool
def get_weather(city: str) -> str:
    """Get the weather for a city."""  # ← LLM reads this!
    return f"The weather in {city} is sunny."
```

The docstring becomes the **tool description** that the LLM uses to decide **when** to call the tool. A bad or missing docstring = LLM won't know when to use your tool!

### Why Type Hints Matter

```python
def get_weather(city: str) -> str:
#                    ^^^
#                    This tells the LLM the argument must be a string
```

Type hints are used to generate the **JSON schema** for the tool's arguments. Without them, the LLM wouldn't know what type of data to pass.

---

## 📊 `bind_tools()` vs `create_react_agent()`

| Feature | `bind_tools()` | `create_react_agent()` |
|---|---|---|
| **What it does** | Tells model about tools | Creates full agent with loop |
| **Tool execution** | Manual (you write the loop) | Automatic |
| **Multiple tool calls** | You handle iteration | Handled automatically |
| **From package** | `langchain` | `langgraph` |
| **Best for** | Learning / custom control | Production / simplicity |

## 🧪 What Happens with Different Inputs?

```python
# ✅ Weather question → LLM calls get_weather
model_with_tools.invoke("What's the weather in Tokyo?")
# → tool_calls: [{'name': 'get_weather', 'args': {'city': 'Tokyo'}}]

# ✅ Non-weather question → LLM responds directly (no tool call)
model_with_tools.invoke("What is 2 + 2?")
# → content: "2 + 2 equals 4" (no tool_calls)

# ✅ Multiple cities → LLM may generate multiple tool calls
model_with_tools.invoke("Compare weather in New York and London")
# → tool_calls: [
#     {'name': 'get_weather', 'args': {'city': 'New York'}},
#     {'name': 'get_weather', 'args': {'city': 'London'}}
#   ]
```

The LLM is smart enough to:
- **Only call tools when relevant** to the user's question
- **Extract the right arguments** from natural language
- **Generate multiple tool calls** when needed

---

## 📝 Summary

1. **`@tool` decorator** converts a function into a tool with schema (name, description, args)
2. **`bind_tools()`** attaches the tool schema to the LLM so it knows what's available
3. **LLM reads** the tool schema + user message and decides whether to call a tool
4. **LLM generates** structured JSON with function name + extracted arguments
5. **Framework executes** the function and passes the result back to the LLM
6. **LLM generates** the final natural language response
7. **`create_react_agent()`** from LangGraph automates the entire loop (steps 3-6)

> **The magic is NOT in string parsing — it's in the LLM's ability to understand natural language and map it to structured tool calls.**
