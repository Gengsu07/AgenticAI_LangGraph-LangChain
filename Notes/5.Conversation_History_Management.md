## 1. Konsep Dasar

### Mengapa Conversation History Penting?

LLM itu **stateless** — setiap request dianggap baru. Jika kita tidak mengirim ulang percakapan sebelumnya, LLM tidak akan "ingat" apa-apa.

```
❌ Tanpa History:
User: "Nama saya Budi"       → AI: "Halo Budi!"
User: "Siapa nama saya?"     → AI: "Saya tidak tahu nama kamu."

✅ Dengan History:
User: "Nama saya Budi"       → AI: "Halo Budi!"
User: "Siapa nama saya?"     → AI: "Nama kamu Budi!"
  ↑ (dikirim semua pesan sebelumnya ke LLM)
```

### Arsitektur Alur Data

```
┌─────────────┐     ┌──────────────────┐     ┌─────────┐
│  User Input │────▶│  History Manager  │────▶│   LLM   │
└─────────────┘     │   (load + save)   │     └────┬────┘
                    └──────────────────┘          │
                           ▲                      │
                           │    ┌─────────────────┘
                           │    ▼
                    ┌──────────────────┐
                    │   Storage Layer  │
                    │ (Memory / DB)    │
                    └──────────────────┘
```

## 2. MessagesPlaceholder vs. Static Messages

### Static Message (Satu Pesan Tetap)

```python
prompt = ChatPromptTemplate.from_messages([
    ("system", "You are helpful."),   # 1 fixed system message
    ("human", "{question}")           # 1 fixed human message
])

# Input: String
chain.invoke({"question": "Hello"})
```

### MessagesPlaceholder (Slot Dinamis untuk Banyak Pesan)

```python
prompt = ChatPromptTemplate.from_messages([
    ("system", "You are helpful."),
    MessagesPlaceholder(variable_name="messages"),  # 0..N messages
])

# Input: LIST of Message Objects
chain.invoke({
    "messages": [
        HumanMessage(content="Hi, I'm Budi"),
        AIMessage(content="Hello Budi!"),
        HumanMessage(content="What's my name?"),
    ]
})
```

| Aspek | Static Message | MessagesPlaceholder |
|:---|:---|:---|
| **Jumlah** | Tepat 1 pesan | 0 sampai N pesan |
| **Input** | String | List of Message Objects |
| **Use Case** | System prompt, single question | Chat history, agent scratchpad |

## 3. In-Memory History

### Cara Kerja

Riwayat disimpan di Python dictionary — hilang saat program restart.

```python
from langchain_community.chat_message_histories import ChatMessageHistory
from langchain_core.chat_history import BaseChatMessageHistory
from langchain_core.runnables.history import RunnableWithMessageHistory

# === STEP 1: Setup Storage (Dictionary) ===
store = {}

def get_session_history(session_id: str) -> BaseChatMessageHistory:
    """Ambil atau buat history berdasarkan session_id."""
    if session_id not in store:
        store[session_id] = ChatMessageHistory()
    return store[session_id]

# === STEP 2: Buat Chain ===
from langchain_core.prompts import ChatPromptTemplate, MessagesPlaceholder
from langchain_groq import ChatGroq

model = ChatGroq(model="Gemma2-9b-It")

prompt = ChatPromptTemplate.from_messages([
    ("system", "You are a helpful assistant. Answer in {language}."),
    MessagesPlaceholder(variable_name="messages"),
])

chain = prompt | model

# === STEP 3: Wrap dengan Message History ===
with_message_history = RunnableWithMessageHistory(
    chain,
    get_session_history,
    input_messages_key="messages",   # key mana yang berisi pesan
)

# === STEP 4: Invoke dengan Config ===
config = {"configurable": {"session_id": "user-budi-001"}}

# Percakapan pertama
response = with_message_history.invoke(
    {"messages": [HumanMessage(content="Hi, nama saya Budi")], "language": "Indonesian"},
    config=config,
)
print(response.content)  # "Halo Budi!"

# Percakapan kedua (history otomatis di-load)
response = with_message_history.invoke(
    {"messages": [HumanMessage(content="Siapa nama saya?")], "language": "Indonesian"},
    config=config,
)
print(response.content)  # "Nama kamu Budi!"
```

### Kapan Pakai In-Memory?
- ✅ Prototyping & development
- ✅ Chatbot sederhana tanpa persistence
- ✅ Unit testing
- ❌ **JANGAN** dipakai untuk production (hilang saat restart)

## 4. Database-Backed History

### Opsi Database yang Tersedia

| Database | Package | Kelebihan | Cocok Untuk |
|:---|:---|:---|:---|
| **SQLite** | `langchain_community` | Simple, tanpa server | Single user, lokal |
| **PostgreSQL** | `langchain_postgres` | Scalable, ACID | Production, multi-user |

```sql

    id SERIAL PRIMARY KEY,
    session_id TEXT NOT NULL,
    message JSONB NOT NULL,
    created_at TIMESTAMP DEFAULT NOW()
);
CREATE INDEX idx_session_id ON chat_history(session_id);
```

### 4C. Redis (Untuk High-Performance)

```python
# pip install langchain-community redis

from langchain_community.chat_message_histories import RedisChatMessageHistory

def get_session_history(session_id: str):
    return RedisChatMessageHistory(
        session_id=session_id,
        url="redis://localhost:6379",
        ttl=3600,  # Auto-expire setelah 1 jam (opsional)
    )

with_message_history = RunnableWithMessageHistory(
    chain,
    get_session_history,
    input_messages_key="messages",
)
```

**Kapan pakai Redis?**
- Chat yang butuh response time sangat cepat
- High-concurrency (banyak user bersamaan)
- Bisa dikombinasi: Redis (short-term cache) + PostgreSQL (long-term archive)

## 5. Strategi Trimming yang Efisien

### Mengapa Perlu Trimming?

```
Problem:
- LLM punya CONTEXT WINDOW terbatas (misal 8192 tokens)
- Setiap percakapan menambah token
- Jika history terlalu panjang → ERROR atau mahal

Solusi:
- Potong (trim) messages yang lama
- Pertahankan yang penting
```

### Strategi 1: Token-Based Trimming (Recommended ✅)

Potong berdasarkan jumlah token — paling akurat karena selaras dengan limit LLM.

```python
from langchain_core.messages import (
    SystemMessage, HumanMessage, AIMessage, trim_messages
)

# Konfigurasi trimmer
trimmer = trim_messages(
    max_tokens=200,          # Maksimum token yang dipertahankan
    strategy="last",         # Pertahankan pesan TERAKHIR (terbaru)
    token_counter=model,     # Gunakan model untuk hitung token
    include_system=True,     # SELALU pertahankan system message
    allow_partial=False,     # Jangan potong pesan di tengah
    start_on="human",        # Mulai dari HumanMessage (bukan AI)
)

# Contoh messages
messages = [
    SystemMessage(content="You're a helpful assistant"),
    HumanMessage(content="Hi! I'm Bob"),
    AIMessage(content="Hi Bob!"),
    HumanMessage(content="I like vanilla ice cream"),
    AIMessage(content="Nice!"),
    HumanMessage(content="What's 2 + 2?"),
    AIMessage(content="4"),
    HumanMessage(content="Thanks"),
    AIMessage(content="No problem!"),
    HumanMessage(content="Having fun?"),
    AIMessage(content="Yes!"),
]

# Trim!
trimmed = trimmer.invoke(messages)
# Result: System message + ONLY the last few messages that fit in 200 tokens
# Note: "Hi! I'm Bob" mungkin hilang → model lupa nama Bob!
```

### Strategi 2: Count-Based Trimming

Potong berdasarkan jumlah pesan — lebih simple tapi kurang akurat.

```python
trimmer = trim_messages(
    max_tokens=6,              # Maksimum 6 pesan (bukan token!)
    strategy="last",
    token_counter=len,         # Hitung berdasarkan jumlah pesan
    include_system=True,
)
```

### Strategi 3: Sliding Window (Manual Implementation)

```python
def sliding_window_history(messages, window_size=10):
    """Pertahankan system message + N pesan terakhir."""
    system_msgs = [m for m in messages if isinstance(m, SystemMessage)]
    non_system = [m for m in messages if not isinstance(m, SystemMessage)]
    
    # Ambil pesan terakhir sebanyak window_size
    recent = non_system[-window_size:]
    
    return system_msgs + recent
```

### Mengintegrasikan Trimmer ke Chain

```python
from operator import itemgetter
from langchain_core.runnables import RunnablePassthrough

chain = (
    RunnablePassthrough.assign(
        messages=itemgetter("messages") | trimmer  # Trim sebelum ke prompt
    )
    | prompt
    | model
)

# Wrap dengan history
with_message_history = RunnableWithMessageHistory(
    chain,
    get_session_history,
    input_messages_key="messages",
)
```

## 6. Advanced: Summarization Strategy

### Problem dengan Trimming Biasa

```
Trimming menghapus pesan lama → informasi penting bisa hilang!

Contoh: User bilang namanya "Budi" di pesan pertama,
tapi setelah 50 pesan, info itu sudah di-trim → model lupa.
```

### Solusi: Summarize + Trim (Hybrid Approach)

```python
from langchain_core.messages import SystemMessage, HumanMessage
from langchain_core.prompts import ChatPromptTemplate

# === STEP 1: Summarizer Chain ===
summarize_prompt = ChatPromptTemplate.from_messages([
    ("system", "Summarize the following conversation into key facts. "
               "Include: user name, preferences, important context. "
               "Be concise but complete."),
    MessagesPlaceholder(variable_name="messages"),
])
summarize_chain = summarize_prompt | model

# === STEP 2: Function to manage history ===
def smart_history_manager(messages, max_messages=20):
    """
    Jika history > max_messages:
    1. Summarize pesan lama
    2. Ganti dengan summary sebagai SystemMessage
    3. Pertahankan pesan terbaru
    """
    if len(messages) <= max_messages:
        return messages  # Belum perlu trim

    # Pisahkan system messages
    system_msgs = [m for m in messages if isinstance(m, SystemMessage)]
    conversation = [m for m in messages if not isinstance(m, SystemMessage)]
    
    # Pesan yang akan di-summarize vs dipertahankan
    cutoff = len(conversation) - max_messages
    old_messages = conversation[:cutoff]
    recent_messages = conversation[cutoff:]
    
    # Generate summary dari pesan lama
    summary = summarize_chain.invoke({"messages": old_messages})
    
    # Gabungkan: System + Summary + Recent
    summary_message = SystemMessage(
        content=f"Summary of earlier conversation:\n{summary.content}"
    )
    
    return system_msgs + [summary_message] + recent_messages
```

### Diagram Alur Summarization

```
Conversation (50 messages):
┌──────────────────────────────────────────────────────┐
│ Msg 1-30 (OLD)              │ Msg 31-50 (RECENT)     │
│ "Hi I'm Budi"               │ "What about Python?"    │
│ "I work at Company X"       │ "Tell me about Django"  │
│ "I like Python"             │ ...                     │
│ ...                         │                         │
└──────────────┬───────────────┴────────────────────────┘
               │
               ▼ Summarize
┌──────────────────────────────────────────────────────┐
│ Summary: "User Budi, works  │ Msg 31-50 (RECENT)     │
│ at Company X, likes Python" │ (kept as-is)           │
│ (1 SystemMessage)           │                         │
└──────────────────────────────────────────────────────┘
Total: 1 summary + 20 recent = 21 messages (dari 50!)
```

## 7. Best Practices

### 🏗️ Architecture

| Practice | Detail |
|:---|:---|
| **Session ID yang Unik** | Gunakan `user_id + timestamp` atau UUID. Jangan hardcode "chat1". |
| **Pisahkan Concerns** | Storage, trimming, dan prompting harus modular. |
| **Multi-Tenant Ready** | Satu user = satu session. Jangan campur history antar-user. |

### 💾 Storage

| Practice | Detail |
|:---|:---|
| **Dev → In-Memory** | Untuk prototyping gunakan `ChatMessageHistory`. |
| **Staging → SQLite** | Untuk testing dengan persistence. |
| **Production → PostgreSQL/Redis** | Untuk scalability dan reliability. |
| **Set TTL di Redis** | Auto-expire session setelah X jam/hari. |
| **Backup Regularly** | Terutama PostgreSQL — schedule pg_dump. |

### ✂️ Trimming

| Practice | Detail |
|:---|:---|
| **Selalu `include_system=True`** | System prompt harus selalu ada. |
| **Gunakan `start_on="human"`** | Jangan mulai dari AIMessage — konteks jadi aneh. |
| **`allow_partial=False`** | Jangan potong pesan di tengah kalimat. |
| **Token-based > Count-based** | Token-based lebih akurat karena sesuai limit LLM. |
| **Pertimbangkan Summarization** | Untuk percakapan panjang yang info lamanya penting. |

### 🔐 Security

| Practice | Detail |
|:---|:---|
| **Jangan log API keys** | Pastikan `.env` di-gitignore. |
| **Sanitize user input** | Hindari prompt injection di chat history. |
| **Encrypt sensitive data** | Terutama jika menyimpan PII di database. |

### 📊 Monitoring

| Practice | Detail |
|:---|:---|
| **Track token usage** | Monitor `response.usage_metadata` untuk cost control. |
| **Log session lengths** | Alert jika session terlalu panjang. |
| **Set max history size** | Hard limit untuk menghindari runaway costs. |

## 8. Quick Reference Cheat Sheet

### Setup Cepat (Copy-Paste Ready)

```python
# ============================================
# PRODUCTION-READY CHATBOT WITH HISTORY
# ============================================

import os
from dotenv import load_dotenv
from operator import itemgetter

from langchain_core.prompts import ChatPromptTemplate, MessagesPlaceholder
from langchain_core.messages import HumanMessage, trim_messages
from langchain_core.runnables import RunnablePassthrough
from langchain_core.runnables.history import RunnableWithMessageHistory

# For SQLite persistence:
from langchain_community.chat_message_histories import SQLChatMessageHistory

# --- Config ---
load_dotenv()

# --- Model ---
from langchain_groq import ChatGroq
model = ChatGroq(model="Gemma2-9b-It")

# --- Trimmer ---
trimmer = trim_messages(
    max_tokens=300,
    strategy="last",
    token_counter=model,
    include_system=True,
    allow_partial=False,
    start_on="human",
)

# --- Prompt ---
prompt = ChatPromptTemplate.from_messages([
    ("system", "You are a helpful assistant. Answer in {language}."),
    MessagesPlaceholder(variable_name="messages"),
])

# --- Chain (dengan trimming) ---
chain = (
    RunnablePassthrough.assign(
        messages=itemgetter("messages") | trimmer
    )
    | prompt
    | model
)

# --- History (SQLite) ---
def get_session_history(session_id: str):
    return SQLChatMessageHistory(
        session_id=session_id,
        connection="sqlite:///chat_history.db"
    )

# --- Final Runnable ---
chatbot = RunnableWithMessageHistory(
    chain,
    get_session_history,
    input_messages_key="messages",
)

# --- Usage ---
config = {"configurable": {"session_id": "user-001"}}

response = chatbot.invoke(
    {"messages": [HumanMessage(content="Hello!")], "language": "English"},
    config=config,
)
print(response.content)
```

### Decision Flowchart

```
Butuh conversation history?
│
├── Tidak → Pakai chain biasa (prompt | model)
│
└── Ya
    │
    ├── Prototyping? → ChatMessageHistory (in-memory)
    │
    └── Production?
        │
        ├── Single user/local? → SQLite
        │
        ├── Multi-user/scalable? → PostgreSQL
        │
        └── High-speed caching? → Redis
            │
            └── History bisa panjang?
                │
                ├── Info lama TIDAK penting → trim_messages (token-based)
                │
                └── Info lama PENTING → Summarize + Trim (hybrid)
```
